# SDD Workflow Init

Analyze a project and create a project-specific spec planning template pre-populated with detected domain context, toolchain, testing patterns, and coding conventions.

**Input**: `/sdd-workflow-init`

No arguments. Runs in the current project directory.

**Output**: `.claude/sdd-workflow/spec-planning-template.md`

---

## Step 1: Scan project sources

Read these in parallel using an Explore agent:

**Identity files:**
- `README.md` (or `readme.md`, `Readme.md`)
- `CLAUDE.md`, `.claude/CLAUDE.md`, `.claude/settings.json`
- `.copilot-instructions.md`
- `CONTRIBUTING.md`, `DEVELOPMENT.md`
- `docs/` directory listing (one level)

**Build and dependency files:**
- `package.json`, `tsconfig.json` (Node/TypeScript)
- `pyproject.toml`, `setup.py`, `setup.cfg`, `requirements.txt` (Python)
- `Cargo.toml` (Rust)
- `go.mod` (Go)
- `Makefile`, `justfile`, `Taskfile.yml`
- Monorepo: `pnpm-workspace.yaml`, `lerna.json`, `nx.json`, `turbo.json`

**Configuration files:**
- `.env.example`, `.env.template` (environment patterns)
- `docker-compose.yml`, `Dockerfile`
- CI config: `.github/workflows/*.yml`, `.gitlab-ci.yml`, `Jenkinsfile`
- Linter config: `.eslintrc*`, `ruff.toml`, `pyproject.toml [tool.ruff]`, `.clippy.toml`
- Type-check config: `tsconfig.json`, `mypy.ini`, `pyrightconfig.json`

**Source structure:**
- Top-level directory listing
- `src/` or `lib/` listing (one level deep)
- Test directory: `tests/`, `test/`, `__tests__/`, `spec/` (structure only)

**Do NOT read:**
- Every source file (scan structure, not content)
- `.env` files (may contain secrets)
- `node_modules/`, `venv/`, `.git/`, or other generated directories

---

## Step 2: Extract project profile

From the scanned sources, identify:

### Language and framework
- Primary language(s) and version constraints
- Frameworks (web, CLI, library, data pipeline, etc.)
- Key dependencies and their purpose (from dependency files)

### Toolchain
| Tool | How detected | Command |
|------|-------------|---------|
| Package manager | dependency file type | `npm`, `pip`, `cargo`, etc. |
| Build | scripts in package.json, Makefile targets | `npm run build`, `make`, etc. |
| Test | test framework in dependencies + config | `pytest`, `jest`, `go test`, etc. |
| Lint | linter config files or scripts | `eslint`, `ruff`, `clippy`, etc. |
| Type-check | type-checker config or scripts | `tsc`, `mypy`, `pyright`, etc. |
| Entry point | main field, scripts, or conventional files | `src/index.ts`, `main.py`, etc. |

### Architecture patterns
- Module organization (monorepo, flat, layered, hexagonal, etc.)
- Key patterns visible in structure (provider pattern, middleware, plugins, MVC, etc.)
- API boundaries (REST, GraphQL, CLI, library, gRPC, etc.)

### Domain context
- What the project does (from README, first paragraph)
- Key domain terminology (from README, docs, module names)
- Domain-specific constraints mentioned in docs or CLAUDE.md

### Conventions
- Naming conventions (from file names, module structure)
- Error handling patterns (from CLAUDE.md, CONTRIBUTING.md)
- Configuration patterns (env vars, config files, feature flags)
- Coding standards documented in CLAUDE.md or CONTRIBUTING.md

## Guardrails

- **Don't over-detect.** If something isn't clear from project sources, leave it as a placeholder. Wrong context is worse than no context.
- **Don't read every source file.** Scan structure and representative files, not the entire codebase. This is a quick setup, not an audit.
- **Respect .gitignore.** Don't read or reference content from gitignored files (they may contain secrets).
- **Ask before overwriting.** If the project template already exists, show what's different and let the user decide.
- **The template is a starting point.** Tell the user to edit it — the detected values are best guesses, not gospel.
